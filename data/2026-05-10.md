### [1] [Is this Build Failure Related to my Patch? An Empirical Study of Unrelated Build Failures in Continuous Integration](https://arxiv.org/abs/2605.05564)
*Andie Huang,Daniel Alencar da Costa,Grant Dick,Mariam El Mezouar*

Main category: cs.SE

TL;DR: 研究Apache开源项目中的77,354个CI构建失败案例，发现开发者平均花费4小时判断失败是否与自己的代码推送相关。通过分析371个无关失败案例，发现20%的无关失败由测试失败引起。构建了基于PU学习的预测模型，F1分数达0.44-0.91，可帮助开发者快速识别无关构建失败。


<details>
  <summary>Details</summary>
Motivation: CI系统中并发的构建失败可能与当前代码推送无关，开发者需要花费大量时间（中位数4小时）来排查失败是否与自己相关，这种时间浪费亟需自动化方法来解决。

Method: 从7个Apache项目中收集77,354个构建失败案例，对10,316个潜在无关失败中抽样371个进行文档分析。提取问题报告、评论和提交相关的33个特征，构建半监督PU学习模型进行无关构建失败预测。

Result: PU学习模型在7个Apache项目上达到精确率0.70-0.88，召回率0.30-1.00，F1分数0.44-0.91，AUC 0.63-0.97。特征重要性分析显示CI延迟、重复错误信息和前置评论数量是关键指标。

Conclusion: PU学习可以有效帮助开发者识别与当前代码推送无关的构建失败，减少不必要的问题排查时间。

Abstract: Continuous Integration (CI) systems often run many builds concurrently. In this setting, a legitimate build failure may not be caused by the code push that triggered it. Such unrelated build failures can waste developer effort because developers must determine whether the failure is actionable for their current change. We study 77,354 CI build failures from seven open source Apache projects to understand and predict unrelated build failures. We find that developers spend a median of 4 hours identifying whether a failure is related or unrelated to their push. We also perform a document analysis of 371 confirmed unrelated build failures sampled from 10,316 potentially unrelated failures. The analysis shows that unrelated test failures account for 20% of the cases in which developers classify build failures as unrelated. To predict unrelated build failures, we extract 33 features from issue reports, issue comments, and commits associated with the triggering push. We build semi-supervised Positive and Unlabeled (PU) learning models for seven Apache projects. The models achieve precision from 0.70 to 0.88, recall from 0.30 to 1.00, F1-score from 0.44 to 0.91, and AUC from 0.63 to 0.97. Feature importance analysis shows that CI latency, repeated error messages, and the number of preceding comments are useful indicators of unrelated build failures. These results show that PU learning can help developers identify build failures that are unlikely to be caused by their current push.

### [2] [SiblingRepair: Sibling-Based Multi-Hunk Repair with Large Language Models](https://arxiv.org/abs/2605.06209)
*Xinyu Liu,Jiayu Ren,Yusen Wang,Qi Xin,Xiaoyuan Xie,Jifeng Xuan*

Main category: cs.SE

TL;DR: SiblingRepair是一种基于LLM的多hunk自动程序修复技术，专门针对代码中相似错误位置的修复。它通过token和嵌入匹配搜索语义相关的sibling候选，利用LLM识别与失败相关的sibling，并通过同步修复和迭代修复两种策略生成一致补丁。在Defects4J和GHRB基准测试中，SiblingRepair显著优于包括Hercules在内的现有方法。


<details>
  <summary>Details</summary>
Motivation: 现有sibling修复技术（如Hercules）受限于对sibling位置的强假设、依赖commit历史、僵化的AST匹配和模板化补丁生成，需要更灵活通用的修复方法。

Method: 1) 基于token和嵌入的代码匹配搜索语义相关的sibling候选；2) 使用LLM识别与失败相关的sibling；3) 两种互补策略：同步修复（联合修复sibling）和迭代修复（逐步分析候选生成补丁）；4) 保留早期可疑位置的补丁，组合成通用多hunk补丁。

Result: 在Defects4J和GHRB基准上显著优于SOTA多hunk修复技术（包括Hercules），修复效率高，sibling检测和修复组件有效，LLM数据泄露对结果影响有限。

Conclusion: SiblingRepair通过LLM驱动的灵活策略有效推进了sibling修复和通用多hunk自动程序修复。

Abstract: Developers often make similar mistakes across code locations implementing related functionalities. These locations, called siblings, share similar issues and require similar fixes. Accurately identifying siblings and consistently repairing them are crucial for automated program repair. Hercules is a SOTA technique designed for sibling repair. However, it is limited by strong assumptions about sibling locations and commit-history availability, rigid AST-based sibling matching, and inflexible template-based patch generation. To address these limitations, we present SiblingRepair, a new LLM-based multi-hunk APR technique specialized for sibling repair. Starting from a suspicious location identified by spectrum-based fault localization, SiblingRepair searches for semantically related sibling candidates using token- and embedding-based code matching, without restricting discovery to failing-test coverage or commit history. It then uses an LLM to identify failure-relevant siblings and generate consistent patches through two complementary strategies: simultaneous repair, which jointly repairs siblings, and iterative repair, which progressively analyzes candidates for patch construction. SiblingRepair further preserves promising patches generated from earlier suspicious locations and combines them into generalized multi-hunk patches. We evaluate SiblingRepair on the Defects4J and GHRB benchmarks. The results show that SiblingRepair substantially outperforms SOTA multi-hunk repair techniques including Hercules. Our evaluation further demonstrates its repair efficiency, the effectiveness of its sibling detection and repair components, and limited impact of the LLM data leakage on the results. Overall, SiblingRepair advances automated sibling and general multi-hunk repair.
